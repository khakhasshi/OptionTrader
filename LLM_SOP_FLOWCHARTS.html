QQQ 日内波动率交易系统：LLM 参与 SOP 流程图

本页面与系统设计、外部接口计划和开发计划使用同一架构。LLM 用于上下文解释、SOP 审核、 持仓归因和研究假设，不拥有交易授权，也不位于止损、撤单、kill switch 或平仓的关键路径。PostgreSQL 是生产系统唯一主数据库，Parquet 只保存高频行情与回放文件。

系统定位 交易驾驶舱 + 半自动信号系统
LLM 角色 解释、审核、归因、改进建议
交易权限 MANUAL_CONFIRM 需人工确认；所有执行均需 Final Risk
硬边界 LLM 和 React 均不能覆盖 Rust Risk & Execution Gateway
主数据库 PostgreSQL；Parquet 仅用于行情与回放

1. LLM 参与边界

核心原则：规则系统负责判断条件是否满足，LLM 负责解释原因、识别冲突、提示风险，人负责确认执行。
LLM 可以做
  • 解释市场状态变化
  • 审核候选交易是否符合 SOP
  • 归因盈亏来自方向、IV、Theta 或滑点
  • 生成复盘和规则改进假设
LLM 不可以做
  • 绕过硬风控
  • 直接扩大仓位
  • 在 No Trade 状态下强行开仓
  • 把新闻情绪当成确定交易信号
程序必须做
  • 结构化输入输出
  • 记录每次 LLM 建议
  • 把规则变更放入回测或模拟
  • MANUAL_CONFIRM 模式保留人工确认
  • 提交 Broker 前执行最终硬风控

4. 候选交易与下单时序图（PAPER / MANUAL_CONFIRM）

该图展开人工确认模式的实际实现边界：React 不直连 Broker；Python 编排策略和 LLM；Rust 执行两阶段硬风控、幂等下单和对账。其他模式在主流程图中定义。 

sequenceDiagram
    autonumber
    participant Trader as 交易员
    participant Web as React Cockpit
    participant API as Python Application API
    participant LLM as LLM Intelligence Layer
    participant Market as Rust Market Core
    participant Risk as Rust Risk & Execution Gateway
    participant Broker as Selected Broker
    participant Audit as PostgreSQL Audit/Review Store

    Market-->>API: HEALTHY Market/Option Snapshot
    API->>API: 生成 Regime/Vol/Signal/CandidateTradePlan
    API->>Risk: EvaluateRisk(plan, phase=INITIAL)
    Risk->>Risk: 校验 DataHealth、事件、仓位、日亏损、broker_id
    Risk-->>Audit: Initial RiskDecision

    alt Initial Risk 拒绝
        Risk-->>API: RiskRejected
        API-->>Web: No Trade + 拒绝原因
    else Initial Risk 通过
        Risk-->>API: InitialRiskPassed
        API->>LLM: 结构化上下文 + plan hash
        alt LLM 有效返回
            LLM-->>API: Proceed / Wait / Cancel / Reduce Risk
        else LLM 超时、不可用或 Schema 错误
            API->>Audit: Review Unavailable
            API->>API: 继续确定性流程，不授予交易权限
        end

        alt Cancel / Conflict
            API-->>Web: 取消或等待
            API->>Audit: 记录未交易信号
        else Reduce Risk
            API->>API: 创建新 plan_id 与 plan hash
            API->>Risk: 新计划重新执行 Initial Risk
        else Proceed / Review Unavailable
            API-->>Web: 展示计划、最大损失、TTL 和审核状态
            Trader->>Web: 人工确认
            Web->>API: confirmed plan hash
            API->>Audit: 锁行并 claim 加密 confirmation capability + 记录确认意图
            Audit-->>API: capability ciphertext（API 内解密）
            API->>Risk: SubmitApprovedPlan(token + plan hash)
            Risk->>Market: 获取最新 DataHealth 和 quote
            Market-->>Risk: 最新不可变快照
            Risk->>Broker: 查询账户、持仓、订单和 buying power
            Broker-->>Risk: BrokerSnapshot + BrokerHealth
            Risk->>Risk: Final Risk Check

            alt Final Risk 拒绝
                Risk-->>Audit: Final RiskRejected
                Risk-->>API: 拒绝提交
                API-->>Web: 计划失效或风险状态变化
            else Final Risk 通过
                Risk->>Broker: 幂等提交唯一限价单
                Broker-->>Risk: Accepted / Rejected / OrderEvent
                Risk-->>Audit: 订单与风险审计
                Risk-->>API: 订单状态
                API-->>Web: Submitted / Working / Partial / Filled
            end
        end
    end

5. 持仓监控与退出时序图

确定性风险判断始终先执行。LLM 只接收审计快照做异步解释，失败时不影响止损、撤单、kill switch 或平仓。 

sequenceDiagram
    autonumber
    participant Market as Rust Market Core
    participant Broker as Selected Broker
    participant Risk as Rust Risk & Execution Gateway / Position Engine
    participant API as Python Application API
    participant LLM as LLM Intelligence Layer
    participant Web as React Cockpit
    participant Trader as 交易员
    participant Audit as PostgreSQL Audit/Review Store

    Market-->>Risk: 最新 quote/IV/spread/DataHealth
    Broker-->>Risk: Order/Fill/PositionSnapshot
    Risk->>Risk: 先评估止损、止盈、时间退出、事件和 kill switch

    alt 触发退出
        alt CONTROLLED_AUTO 或已授权保护退出
            Risk->>Broker: 幂等提交退出/减仓限价单
            Broker-->>Risk: OrderEvent / FillEvent
            Risk-->>Audit: 退出决策与成交审计
        else MANUAL_CONFIRM
            Risk-->>API: MandatoryClosePlan + 原因 + TTL
            API-->>Web: 高优先级退出确认
            Trader->>Web: 确认退出
            Web->>API: confirmed close-plan hash
            API->>Audit: claim 加密退出 capability + 记录确认意图
            API->>Risk: ClosePosition(token + close-plan hash)
            Risk->>Risk: Final Risk Check for Close
            Risk->>Broker: 提交退出/减仓限价单
            Broker-->>Risk: OrderEvent / FillEvent
            Risk-->>Audit: 退出决策与成交审计
        end
    else 未触发退出
        Risk-->>Audit: 持仓状态与未退出原因
    end

    Risk-->>API: 只读 Position/Risk Attribution Snapshot
    API->>LLM: 异步提交结构化归因上下文
    alt LLM 有效返回
        LLM-->>API: 方向/IV/Theta/滑点解释
        API-->>Web: 展示解释，不改变 RiskDecision
    else LLM 不可用
        API->>Audit: Review Unavailable
        API-->>Web: 仅展示确定性持仓与风险状态
    end

6. 规则改进验证流程图

LLM 只能生成研究假设。任何规则改动必须依次通过成本回测、walk-forward、样本外、shadow/paper 和人工 Gate Review。 

flowchart LR
    A[PostgreSQL Daily Review
信号/交易/未交易/SOP违规] --> B[LLM 生成规则改进假设]
    B --> C[Research Queue
人工定义可检验假设]
    C --> D[Backtest
bid-ask/手续费/滑点/TTL]
    D --> E[Walk-forward + 样本外验证]
    E --> F{指标和稳定性通过?}
    F -->|否| G[归档或修改假设]
    F -->|是| H[Shadow / Paper Trading]
    H --> I{运行门槛通过?}
    I -->|否| G
    I -->|是| J[人工 Gate Review]
    J --> K{批准?}
    K -->|否| G
    K -->|是| L[创建新 rule_version
记录 checksum 和报告]
    L --> M[下一交易会话生效]

7. LLM 输出标准

所有 LLM 输出必须是结构化结果，方便系统记录、审计和后续验证。

{
  "schema_version": "1.0",
  "review_id": "llm_review_20260720_094501",
  "plan_id": "plan_20260720_094501",
  "review_status": "COMPLETED | UNAVAILABLE | INVALID",
  "summary": "当前市场状态解释",
  "decision_support": "支持或反对当前候选交易的理由",
